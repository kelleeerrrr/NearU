## 🎯 USER STORIES

### User Story 1: Interactive Map-Based Property Discovery
**As a Customer,** I want to find rental properties using an integrated map with directions and select my requirements via a checklist, so that I can easily visualize the location of my next home and ensure it has all my must-have amenities through a simple interface.

**Acceptance Criteria:**
- An interactive map interface is available on the search page to help users find places and view exact property locations
- The system provides built-in directions and estimated travel times from the user's current location to the listed property
- Instead of traditional dropdown filters, a comprehensive checklist is provided for selecting amenities (e.g., Wi-Fi, Pet-friendly, Parking)
- The map and results list update dynamically in real-time as users check or uncheck items in the checklist
- Each property listing displays a summary of "Checked" amenities to quickly confirm a match with user preferences

---

### User Story 2: Enhanced Search and Filtering Experience
**As a Customer,** I want to use an advanced filtering system that allows me to combine multiple criteria (price range, distance from campus, amenities) and save my search preferences for future visits, so that I can quickly find properties that match my specific needs without scrolling through irrelevant listings.

**Acceptance Criteria:**
- Advanced filtering interface with multiple filter options (price slider, distance radius, amenity checkboxes, gender policy)
- System saves user's search preferences and applies them automatically on future visits
- Filter combinations can be saved as presets for quick access
- Real-time result updates as filters are adjusted
- Clear filter reset option to start new searches
- Mobile-responsive filter panel for easy use on any device

---

### User Story 3: One-Click Visit Scheduling
**As a Customer,** I want to schedule property visits directly from listing details without navigating to multiple pages, so that I can quickly book appointments that fit my schedule and receive instant confirmations.

**Acceptance Criteria:**
- "Schedule Visit" button prominently displayed on each property listing
- Calendar integration shows available time slots in a popup interface
- System suggests optimal visit times based on property owner's availability
- Instant confirmation messages for successful scheduling
- Automatic calendar sync with mobile and email platforms
- Quick reschedule option for confirmed visits

---

### Personalized Dashboard and Recommendations
**As a Customer,** I want a personalized dashboard that shows all the property listings, saved properties, and recommendations based on my preferences, so that I can easily track my housing search journey and discover new properties that match my needs.

**Acceptance Criteria:**
- Dashboard displays saved properties
- AI-powered recommendations based on search history and saved preferences
- Quick access to recently viewed properties for comparison
- Personalized notifications for price changes and new matching properties
- One-click access to saved properties and search history
- Mobile-optimized dashboard layout

---

### User Story 5: Mobile-First Property Management
**As a Property Owner,** I want to manage my property listings through a mobile-first interface that allows me to add, edit, and update properties on the go, with instant photo uploads from my device camera.

**Acceptance Criteria:**
- Progressive Web App design optimized for mobile devices
- Camera integration for direct photo uploads and cropping
- Bulk editing capabilities for multiple listings
- Real-time status updates (available/unavailable) with instant sync
- Offline mode for basic property management
- Push notifications for inquiries and booking requests
- Quick response templates for common customer questions

## 🎯 **STUDENT CORE FUNCTIONALITY** [PRIORITY 1]

### Use Case 1: User Authentication and Registration
**Most Critical Student Function - Foundation for All Student Activities**

### A. Actors
Student, Owner, Admin, System

### B. Preconditions
User is not logged into the system.

### C. Triggers
User clicks on "Login" or "Sign Up" button.

### D. Basic Flow (Main Scenario)
1. User navigates to authentication page
2. User enters email and password (for login) or registration details
3. For registration: User selects account type (Student/Owner)
4. User fills in required information (name, email, phone, password)
5. System validates input data and checks for duplicates
6. System creates new user account with appropriate role
7. System logs user in automatically
8. System redirects to appropriate dashboard

### E. Alternative Flows
- Invalid login credentials: System shows error message with retry option
- Email already exists: System prompts user to login instead
- Password requirements not met: System shows specific validation errors
- Account type selection missing: System prevents registration

### F. Postconditions
User is successfully authenticated and redirected to their respective dashboard with proper permissions.

---

### Use Case 2: Dorm Search and Filtering
**Primary Student Function - Core Discovery System**

### A. Actors
Student, System

### B. Preconditions
Student is logged into the system.

### C. Triggers
Student accesses the home page or uses search functionality.

### D. Basic Flow (Main Scenario)
1. Student enters search terms (street, type, owner name)
2. System displays matching dorm listings with real-time results
3. Student applies advanced filters (price range, distance, gender policy, amenities, availability)
4. System updates results dynamically based on filter selections
5. Student can sort results by price, distance, rating, or availability
6. Student clicks on listing to view detailed information

### E. Alternative Flows
- No results found: System shows "No listings available" with search suggestions
- Invalid filter combinations: System provides helpful error messages
- Network connectivity issues: System displays cached results with notification

### F. Postconditions
Student can find and filter dorm listings based on their specific preferences and requirements.

---

### Use Case 3: Schedule Dorm Visit
**Essential Student Function - Direct Property Interaction**

### A. Actors
Student, Owner, System

### B. Preconditions
Student is logged in and has found a dorm listing they're interested in.

### C. Triggers
Student clicks on "Schedule Visit" button from dorm details page.

### D. Basic Flow (Main Scenario)
1. Student selects preferred visit date from calendar interface
2. Student chooses available time slot from available options
3. Student adds optional notes for the owner
4. Student submits visit request with contact preferences
5. System sends immediate notification to property owner
6. System creates visit record with "Pending" status
7. Confirmation email is sent to both student and owner

### E. Alternative Flows
- Selected date/time is unavailable: System shows available alternatives
- Owner rejects the visit: System updates status to "Rejected" and notifies student
- Student cancels the visit: System updates status to "Cancelled" and notifies owner
- Owner requests reschedule: System allows modification with student confirmation

### F. Postconditions
Visit is scheduled, all parties are notified, and visit appears in student's "Scheduled Visits" list.

---

### Use Case 4: Dorm Comparison
**Key Student Function - Informed Decision Making**

### A. Actors
Student, System

### B. Preconditions
Student has viewed multiple dorm listings.

### C. Triggers
Student clicks "Compare" button on dorm listings.

### D. Basic Flow (Main Scenario)
1. Student selects 2-4 dorms to compare from search results
2. System displays side-by-side comparison table with detailed features
3. System shows key metrics (price, distance, rating, amenities, availability)
4. Student can view detailed differences and highlights
5. Student can save comparison for later review
6. Student can share comparison results

### E. Alternative Flows
- Student tries to compare more than 4 dorms: System limits selection with suggestion
- Comparison data becomes outdated: System refreshes automatically
- Selected dorm becomes unavailable: System updates comparison

### F. Postconditions
Student can make informed decisions by comparing multiple dorm options with comprehensive data.

---

### Use Case 5: Messaging and Inquiry System
**Important Student Function - Communication Channel**

### A. Actors
Student, Owner, System

### B. Preconditions
Both parties are logged into the system.

### C. Triggers
Student clicks "Contact Owner" or owner responds to inquiry.

### D. Basic Flow (Main Scenario)
1. Student sends message about dorm listing with inquiry details
2. System delivers message to owner's inbox with notification
3. Owner receives notification and can view message details
4. Owner replies to student inquiry with additional information
5. System maintains conversation history with timestamps
6. Both parties can exchange contact information if mutually agreed

### E. Alternative Flows
- Student sends inappropriate message: System flags and may restrict account
- Owner doesn't respond within 24h: System sends reminder notification
- Message contains spam or contact info: System filters and warns
- User blocks another user: System prevents further messaging

### F. Postconditions
Communication channel is established between student and property owner with message history preserved.

---

### Use Case 6: Review and Rating System
**Valuable Student Function - Community Feedback**

### A. Actors
Student, Owner, System

### B. Preconditions
Student has completed a dorm visit or stay at the property.

### C. Triggers
Student clicks "Write a Review" or system prompts for feedback after visit completion.

### D. Basic Flow (Main Scenario)
1. Student selects star rating (1-5 stars) for various aspects
2. Student writes detailed review text about their experience
3. Student can upload photos with review for authenticity
4. System submits review to dorm listing with verification
5. System calculates and updates overall dorm rating automatically
6. Owner receives notification of new review and can respond

### E. Alternative Flows
- Student tries to review without verified stay: System blocks action
- Review contains inappropriate content: System flags for moderation
- Duplicate review from same user: System prevents submission
- Review violates platform guidelines: System rejects with explanation

### F. Postconditions
Review is published and contributes to dorm's overall rating with transparency for future students.

### A. Actors
Student, System

### B. Preconditions
Student is logged into the system.

### C. Triggers

### D. Basic Flow (Main Scenario)
1. Student adds dorm to saved list with one-click action
2. System saves preference to user profile with organization options
3. Student can view all saved listings in dedicated "Saved" section
4. Student receives notifications for price changes or availability updates
5. Student can quickly access saved listings for comparison and booking
6. Student can remove saved listings when no longer needed

### E. Alternative Flows
